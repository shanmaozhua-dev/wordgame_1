# 04_命名与目录规范

本文件规定地图、事件、资源、开关和交接文件如何命名。

## 固定目录

```text
Scenes/
  Maps/
    第一章/
    第二章/
    第三章/
    第四章/
    第五章/
    第六章/
    第七章/
    第八章/
    系統/
    測試用/
  Events/
  Animations/
  Player/
  UI/
Scripts/
Docs/
  LevelWorkflow/
```

## 地图命名

保持源项目地图名，不要随意改简体、英文或拼音。

示例：

```text
Scenes/Maps/第一章/02_長廊.tscn
Scenes/Maps/第五章/24_議事廳_說故事.tscn
Scenes/Maps/第七章/09_魔龍_黑洞.tscn
```

`map_transport` 中写法：

```text
"map": "第一章/02_長廊"
```

不要写：

```text
"map": "res://Scenes/Maps/第一章/02_長廊.tscn"
```

## 复刻工程命名

如果复刻工程需要另建场景，仍建议保留源文件名。确实需要英文 ID 时，写在 manifest 的 `level_id`，不要替换文件名。

示例：

```json
{
  "level_id": "ch1_02_corridor",
  "display_name": "第一章 02_長廊",
  "source_map": "D:/文字游戏/Scenes/Maps/第一章/02_長廊.tscn"
}
```

## 事件命名

事件名是命令目标，必须稳定。

示例：

```text
門
進來門
進來間
開始事件
觸發一
觸發二
```

规则：

- 不要为了“看起来整齐”重命名源事件。
- 如果事件名用于 `target`、`copy_from_event`、变量记忆或句子规则，改名会破坏流程。
- 同一地图内事件名必须唯一。
- 运行时生成事件可使用规则名，但也要在 manifest 记录。

## 开关和变量命名

源项目常见形式：

```text
ch1_進長廊
ch1_長廊_開啟塔門
ch4_寶庫穹頂_組字成功
```

规则：

- 保留源项目开关名。
- 新增开关使用 `ch<章节>_<地点>_<状态>`。
- 自开关写在事件逻辑里，不要和全局开关混用。
- 不确定开关含义时，在 handoff 里标注“待确认”，不要改名。

## 章节段落 ID

`enter_section` 常见格式：

```text
@[enter_section] {
  "section_name": "1-2",
  "need_save": true
}
```

规则：

- 使用源项目已有 `section_name`。
- 新增段落时优先跟章节和地图序号对齐。
- 与提示系统、存档和分析数据相关，不要随意删。

## 资源路径

Godot 内部路径使用 `res://`。

外部来源路径使用 `D:/文字游戏/...` 记录在 manifest。

示例：

```text
source: D:/文字游戏/Scenes/Animations/Pull.tscn
target: res://Scenes/Animations/Pull.tscn
```

## Manifest 和 handoff 命名

每个地图交付时，建议放在：

```text
Docs/LevelWorkflow/Handoffs/<level_id>/
  level_manifest.json
  handoff.md
```

`level_id` 使用稳定英文小写：

```text
ch1_02_corridor
ch4_13_treasure_helmet_tutorial
ch7_09_dragon_blackhole
```

## 测试/旧版/临时地图

以下命名通常表示非正式主线地图：

```text
測試用/
_test
_舊
99_
X / XX
PT用
```

这些地图可以分析参考，但不得替代正式地图交付，除非组内明确指定。
